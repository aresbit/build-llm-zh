# 第7章: 指令微调 (Fine-Tuning to Follow Instructions)

我们在第6章实现了对预训练LLM的分类微调（让模型判断垃圾邮件）。本章转向更具通用性的任务：让LLM学会**遵循人类指令**（instruction following），即指令微调（instruction fine-tuning），也称监督指令微调（supervised instruction fine-tuning）。

## 7.1 指令微调概述

预训练LLM经过自回归语言建模训练后，具备文本补全能力——给定前缀，生成后续token。但预训练模型往往无法正确响应明确指令，例如"修正这段文字的语法错误"或"将此句改写为被动语态"。指令微调的目标就是对基座模型进行额外的有监督训练，使其学会根据指令（instruction）生成期望的回答（response）。

指令微调的数据集由若干**(指令, 回答)**对构成，每一条数据形如：

```
{
  "instruction": "Convert 45 kilometers to meters.",
  "input": "",
  "output": "45 kilometers is 45000 meters."
}
```

instruction字段存放指令文本，input字段存放可选的上下文输入，output字段存放期望的回答。

### 指令微调的数学目标

指令微调本质上仍是一个自回归语言建模任务，目标函数与预训练一致——最大化条件似然。但对于结构化(instruction, response)对，训练数据的组织形式有所不同。

设一条完整的训练文本由**指令部分** $$T_{\text{inst}}$$ 和**回答部分** $$T_{\text{resp}}$$ 拼接而成：

$$T = T_{\text{inst}} \oplus T_{\text{resp}}$$

其中 $$\oplus$$ 表示字符串/序列拼接。tokenization后得到token序列 $$x = (x_1, x_2, \ldots, x_n)$$，令 $$n_{\text{inst}}$$ 为指令部分（含格式模板）的token数，则 $$x_{1:n_{\text{inst}}}$$ 为指令token，$$x_{n_{\text{inst}}+1:n}$$ 为回答token。

指令微调的损失函数为：

$$L(\theta) = -\sum_{t = n_{\text{inst}} + 1}^{n} \log P_{\theta}(x_t \vert  x_{<t})$$

**关键的Loss Masking设计**：损失仅在回答token上计算，指令token不参与损失计算。这样做有双重目的：

1. **避免模型死记指令**：如果损失也作用于指令token，模型会不必要地学习复述指令，而非专注于生成高质量回答。
2. **提升泛化能力**：教模型"看到指令后生成回答"，而非"看到指令后复述指令+生成回答"，这更贴合推理时的使用场景。

在PyTorch的实现中，通过将指令部分的目标token ID设为 `-100`（PyTorch交叉熵默认的 `ignore_index`）来屏蔽：

$$\text{targets}[0:n_{\text{inst}} - 1] = -100$$

这样，交叉熵函数 `F.cross_entropy(logits, targets, ignore_index=-100)` 会自动忽略这些位置，计算出的梯度完全来自回答部分的预测误差。

> **注**：2024年Shi等人的论文 *"Instruction Tuning With Loss Over Instructions"* 指出，当指令很长而回答很短、或训练样本很少时，不对指令做mask可能反而有益（参见附录B文献）。本书正文采用mask方案，不做mask的实验留作练习。

## 7.2 构建监督指令微调数据集

### 7.2.1 Alpaca风格的提示模板 (Prompt Template)

不同的指令微调LLM使用不同的提示格式。本书采用Alpaca风格：

```
Below is an instruction that describes a task. Write a response
that appropriately completes the request.

### Instruction:
{instruction}

### Input:
{input}                          # 可选; 为空则省略

### Response:
{output}
```

对应的 `format_input` 函数：

```python
def format_input(entry):
    instruction_text = (
        f"Below is an instruction that describes a task. "
        f"Write a response that appropriately completes the request."
        f"\n\n### Instruction:\n{entry['instruction']}"
    )
    input_text = f"\n\n### Input:\n{entry['input']}" if entry["input"] else ""
    return instruction_text + input_text
```

完整训练文本为 `format_input(entry) + "\n\n### Response:\n" + entry["output"]`。

### 7.2.2 数据划分

本书使用含1100条指令-回答对的自定义数据集，按85%/10%/5%划分为训练集(935条)、测试集(110条)和验证集(55条)。

## 7.3 训练批次的组织

指令微调的批处理过程比分类微调更复杂，因为每条样本长度不一且需要做loss masking。关键步骤为：

### 7.3.1 InstructionDataset类

在 `__init__` 中对每条数据进行**预tokenize**（调用 `tokenizer.encode(full_text)`），存储token ID序列。

### 7.3.2 自定义collate函数

由于各条样本长度不同，需自定义 `custom_collate_fn`。其核心逻辑为：

1. **找到批次内最长序列长度** $$L_{\text{max}} = \max_i (\vert x^{(i)}\vert  + 1)$$
2. **填充至统一长度**：每条序列尾部追加 `<|endoftext|>` token (ID=50256)
3. **构造输入与目标**：
   - 输入：`padded[:-1]`
   - 目标：`padded[1:]` （右移一位，标准的next-token prediction）
4. **替换padding token为ignore_index**：将目标序列中除第一个外的所有50256 token替换为-100

### 7.3.3 损失计算的详细推导：Why -100？

设模型对单个样本的输出logits为 $$Z \in \mathbb{R}^{L \times V}$$（L为序列长度，V为词表大小），目标token ID序列为 $$y \in \mathbb{Z}^{L}$$。标准交叉熵损失为：

$$L = \frac{1}{L} \sum_{t=1}^{L} -\log \frac{e^{Z_{t, y_t}}}{\sum_{v=1}^{V} e^{Z_{t, v}}} = \frac{1}{L} \sum_{t=1}^{L} \ell_t$$

当 $$y_t = -100$$ 时，PyTorch的 `cross_entropy(..., ignore_index=-100)` 会自动跳过该位置：

$$L_{\text{masked}} = \frac{1}{\vert \{t: y_t \neq -100\}\vert } \sum_{t: y_t \neq -100} \ell_t$$

更进一步地，当我们对指令部分也做mask（$$y_t = -100$$ for $$t \leq n_{\text{inst}}$$），最终的有效损失为：

$$L_{\text{inst-tuning}} = \frac{1}{\vert \{t: t > n_{\text{inst}}\}\vert } \sum_{t = n_{\text{inst}} + 1}^{L} -\log P_{\theta}(x_t \vert  x_{<t})$$

## 7.4 创建指令数据集的DataLoader

将 `InstructionDataset` 实例和 `custom_collate_fn`（通过 `functools.partial` 预设 `device` 和 `allowed_max_length=1024`）传入PyTorch的 `DataLoader`类，分别创建训练、验证和测试加载器。

由于 `allowed_max_length=1024`，所有超过GPT-2 1024 token上下文窗口的序列将被截断。自定义collate函数的按批次填充策略（而非全局统一长度）显著减少了不必要的padding。

## 7.5 加载预训练LLM

使用GPT-2 medium（355M参数）而非GPT-2 small（124M）。原因：124M模型容量太小，无法在指令微调中获得令人满意的结果——它缺乏足够的学习和保留复杂指令模式的能力。

```python
CHOOSE_MODEL = "gpt2-medium (355M)"
# emb_dim=1024, n_layers=24, n_heads=16
```

加载后，首先用一条验证集指令测试原始模型的回答质量。预训练模型通常只是机械地重复输入，无法正确理解指令——这正是需要通过微调解决的问题。

## 7.6 对指令数据进行LLM微调

### 7.6.1 训练配置

复用第5章的 `train_model_simple` 函数进行训练。关键超参数：

- **优化器**：AdamW，学习率 $$5 \times 10^{-5}$$
- **Weight decay**：0.1 （详见附录D的AdamW推导）
- **训练周期**：2个epoch
- **评估频率**：每5步评估一次

### 7.6.2 损失计算详情

每次迭代的损失计算流程为：

1. 模型前向传播：$$\text{logits} = \text{model}(\text{inputs}) \in \mathbb{R}^{B \times L \times V}$$
2. 对logits和targets调用 `F.cross_entropy`（内部包含softmax + NLL）
3. masked的padding和instruction token在损失中自动忽略

### 7.6.3 训练结果分析

两个epoch后，训练损失从初始的约3.83降至约0.30，验证损失从约3.76降至约0.66。模型成功学会了将主动语态句子改写为被动语态等指令。

完整的损失曲线（如图7.17所示）显示了初期快速下降后逐渐收敛的典型模式。第二个epoch的损失下降速度放缓，表明模型已经接近学习能力的上限。

## 7.7 回答的提取与保存

为评估模型在测试集上的性能，使用 `generate` 函数对每条测试样本生成回答，然后切片提取纯回答部分：

```python
response_text = generated_text[len(input_text):].replace("### Response:", "").strip()
```

所有测试结果（连同原始指令和期望回答）保存为 `instruction-data-with-response.json`，以供后续评估。训练好的模型权重保存为 `gpt2-medium355M-sft.pth`。

## 7.8 评估微调后的LLM

### 7.8.1 LLM-as-Judge方法

评估指令微调LLM的对话性能比分类准确率复杂得多。实践中常用的三种方法：

1. **选择题基准测试**（如MMLU，https://arxiv.org/abs/2009.03300）：测试模型在各学科的知识水平，由准确率衡量。核心公式（基于多项选择）：

   $$\text{Accuracy} = \frac{\#\text{correct answers}}{\#\text{total questions}}$$

2. **人工偏好比较**（如LMSYS Chatbot Arena，https://arena.lmsys.org）：人类评估者比较不同模型的回答质量，通过ELO评分系统排名：

   $$E_A = \frac{1}{1 + 10^{(R_B - R_A)/400}}$$

   其中 $$R_A$$、$$R_B$$ 分别为两模型的ELO评分。

3. **LLM-as-Judge自动评估**（如AlpacaEval，https://tatsu-lab.github.io/alpaca_eval/）：用一个强大的LLM（如GPT-4或Llama 3）作为"裁判"，对生成回答打分：

   $$\text{score} = \text{JudgeLLM}(\text{instruction}, \text{reference\_output}, \text{model\_response})$$

### 7.8.2 ROUGE和BLEU指标

虽然本章使用LLM-as-Judge，但传统的自动评估指标也可作为参考：

**ROUGE-L**（Recall-Oriented Understudy for Gisting Evaluation）基于最长公共子序列（LCS）：

$$\text{ROUGE-L recall} = \frac{\vert LCS(\text{reference}, \text{candidate})\vert }{\vert \text{reference}\vert }$$

$$\text{ROUGE-L precision} = \frac{\vert LCS(\text{reference}, \text{candidate})\vert }{\vert \text{candidate}\vert }$$

$$\text{ROUGE-L F1} = \frac{2 \cdot \text{ROUGE-L recall} \cdot \text{ROUGE-L precision}}{\text{ROUGE-L recall} + \text{ROUGE-L precision}}$$

**BLEU**（Bilingual Evaluation Understudy）基于n-gram匹配精度，带简短惩罚项（brevity penalty, BP）：

$$\text{BLEU} = \text{BP} \cdot \exp\left(\sum_{n=1}^{N} w_n \log p_n\right)$$

其中 $$p_n$$ 为n-gram匹配精度，$$w_n = 1/N$$ 为均匀权重（通常N=4），BP惩罚过短的候选回答：

$$\text{BP} = \begin{cases} 1 & \text{if } c > r \\ e^{1 - r/c} & \text{if } c \leq r \end{cases}$$

（$$c$$ 为候选回答长度，$$r$$ 为参考回答长度）

### 7.8.3 使用Llama 3进行自动评估

本书使用Ollama在本地运行Llama 3（8B参数指令微调版）作为评估模型。评估prompt为：

```
Given the input `{format_input(entry)}` and correct output `{entry['output']}`,
score the model response `{entry['model_response']}`
on a scale from 0 to 100, where 100 is the best score.
Respond with the integer number only.
```

对110条测试样本进行评分，微调后的GPT-2 medium平均得分约**50.32**（满分100）。作为参考：未微调的Llama 3 8B基座模型平均得分约58.51，指令微调版Llama 3 8B-Instruct得分约82.6。

### 7.8.4 改进方向

- 调整超参数（学习率、batch size、训练周期）
- 扩大训练数据集（如使用完整的52,000条Stanford Alpaca数据集）
- 尝试不同的提示模板（如Phi-3的 `<|user|>` / `<|assistant|>` 格式）
- 使用更大的基座模型
- 在指令微调之后进行偏好微调（preference fine-tuning，如DPO）

## 7.9 总结与展望

本章完成了LLM开发周期的最后一步——指令微调。从第4章的模型架构实现，到第5章的预训练，再到第6章的微调分类和本章的指令微调，我们完整走通了从零构建一个能遵循人类指令的ChatGPT式LLM的全过程。
